# 模板扫描维度定义

被 `Skill: 提取Harness模板` Step 5 引用的验证规则集。非独立 Skill，不单独执行。

空白模板的核心要求：不含任何项目相关内容，所有项目定制点均使用 `{{...}}` 占位符，结构完整可直接用于新项目初始化。

---

## 扫描维度

### 维度1: 模板净化（项目残留检测）

模板中不应存在任何来自源项目的具体内容。检查项：

- 项目名称泄漏：搜索源项目名称（从源项目 PROJECT.md 提取，或由用户提供）在模板文件中的出现
- 领域术语泄漏：非占位符文本中出现的项目特有业务术语、技术栈名称（如特定框架名、API名、特定平台术语），排除 Harness 框架自身术语
- 硬编码路径/文件名：引用了源项目特有的文件名、目录名、plan/spec 文件名
- 项目特有示例：示例文本、示例命令、示例 prompt 中包含源项目的具体场景
- 项目特有规则：扫描维度表、严重程度排序、文件类型映射等直接搬运了源项目配置

判定原则：
- framework/ 下的文件（通用能力）不应包含任何项目特有内容（包括项目特有的扫描维度、严重程度排序、文件类型映射等配置）
- knowledge/、prd/、PROJECT.md 中，`{{...}}` 占位符外部的非 Harness 术语均视为泄漏（即使看似是"格式示例"）
- guides/ 中不应包含源项目的具体任务记录、plan 文件名或执行示例

输出：泄漏内容原文 + 所在文件及行号 + 泄漏类型 + 建议修复

### 维度2: AI可理解性（严重错误检测）

AI 无法理解或执行的内容会导致模板使用时出错。检查项：

- 残留 TODO/FIXME/HACK 注释：模板中的开发备忘不应发布给用户
- 矛盾指令：同一模板内，不同文件对同一规则的描述冲突（如 FRAMEWORK.md vs Workflow 文件）
- 不完整段落：有标题但无内容的空白章节、截断的句子、未闭合的列表
- 损坏的 Markdown 结构：未闭合代码块、格式错乱的表格、断裂的链接语法
- 循环或死路径引用：A 引用 B 引用 A，或指令指向一个不存在的后续步骤
- 编码异常：非预期的特殊字符、不可见字符、混合换行符

输出：问题内容原文 + 所在文件及行号 + 问题类型 + 严重程度（高/中/低） + 建议修复

### 维度3: 占位符完整性

项目定制点必须使用 `{{...}}` 占位符，且占位符格式正确。检查项：

- 缺失占位符：PROJECT.md 和 knowledge/、prd/ 中存在需要项目定制但未使用占位符的内容（对照同类已有占位符推断）
- 占位符语法：`{{` 和 `}}` 是否配对、嵌套是否正确
- 占位符描述：占位符内的说明文字是否足够清晰，能让用户理解需要填入什么
- 假占位符：看似占位符但实际是固定内容（如 `{{固定值不需要改}}`）
- 语法不一致：所有占位符必须统一使用 `{{...}}` 语法，不允许 `${...}` 或其他变体

输出：问题位置 + 问题类型 + 建议修复

### 维度4: 结构完整性

模板目录结构必须与 FRAMEWORK.md 声明一致且完整。检查项：

- 必需文件存在性：CLAUDE.md、AGENTS.md、PROJECT.md、FRAMEWORK.md 及 FRAMEWORK.md 注册表中列出的所有 Agent/Workflow/Skill 文件
- 必需目录存在性：knowledge/、prd/、lessons/、specs/active/、specs/completed/、plans/active/、plans/completed/；空目录必须包含 .gitkeep 文件（git 不跟踪空目录，缺少 .gitkeep 会导致模板分发后目录丢失）
- 元数据完整性：Skill/Workflow/Agent 文件须有 YAML frontmatter（name + description）；knowledge/prd/lessons .md 文件须有首行 `<!-- SUMMARY: ... -->` 注释
- 多余文件：模板中存在但 FRAMEWORK.md 未注册、且非标准结构要求的文件

输出：缺失/多余项 + 期望位置 + 建议修复

### 维度5: 内部引用有效性

模板内的交叉引用必须指向模板内存在的文件。检查项：

- `@引用` 解析：所有 `@.harness/...`、`@AGENTS.md` 等引用的目标文件是否存在于模板中
- 正文路径引用：`.harness/...` 格式的路径是否指向模板内存在的文件
- 表格中的文件路径列：注册表、索引表中的路径是否有效
- 注册同步性：FRAMEWORK.md 和 PROJECT.md 中的注册表条目与模板内实际文件双向一致

输出：失效引用 + 所在文件及行号 + 建议修复

### 维度6: 版本漂移（需提供 source_dir）

当提供源项目路径时，比对模板与源项目的 framework/ 通用文件是否同步。检查项：

- 比对基准：使用源项目工作区当前文件（而非 git HEAD），以捕获最新修改
- 逐文件比对 `{tpl_dir}/.harness/framework/` 与 `{source_dir}/framework/` 下的同名文件
- 标记有差异的文件及差异摘要
- 区分"预期差异"（源项目有项目定制而模板已泛化）和"意外差异"（模板遗漏了源项目的框架改进）
- 检查源项目注册表中新增的条目是否同步到模板注册表

输出：漂移文件列表 + 差异类型（预期/意外） + 差异摘要

### AI自主扩展

上述维度为基线检查项。扫描中发现不属于以上维度但影响模板质量的问题，归入"维度7: 其他发现"报告，并建议是否固化为新维度。
