## 项目迭代开发
1. 定义需求。更新到 `03-prd-specs.md`
2. 提交需求、给AI
    - 推荐方式：引用`03-prd-specs.md`需求字段、到Chat消息框（CC Option+K）。
    - 流程控制：如弱模不遵守流程，可在Chat框前缀Skill，如 `执行Skill 迭代功能，{功能描述}`
3. 确认需求。人工确认AI识别到的需求，产出物如下；如需调整、转回步骤1
    - 执行计划（AI 自主管理在 .harness/specs/、.harness/plans/，跟踪进度和技术债）
4. 实现需求。AI按照Harness规范，大部分自主执行、少量人工确认(如删除文件)。产出物如下
    - 代码（已验收通过）
    - AI知识库（主要是 .harness/knowledge/ 文档库）
5. 验收需求。查看任务报告，按需擦屁股(如补充功能测试)
6. [可选]回填知识。经过多轮迭代后，可以考虑 更新产品知识库、Harness知识库，指令和更新对象如下
    - 提示词：执行Skill `从代码治理知识库`
    - 提示词：执行Skill `回填产品文档`

- 产品

    - 建议周期：周

- 其它
    - 提示词：按照OpenAI HarnessEngineer指导、Anthropic最佳实践，检查项目Harness知识库的RULE、SKILL、Prompt等描述，给出必要的改进意见，让用户确认后才能修改，意见和修改都要按照等级分批次；
    - 提示词：检查Harness知识库，包括 CLAUDE.md、AGENTS.md、.harness/目录下的所有文档，修复错误、过时的描述；如遇AI只读的文档，让用户确认后再修改；

## 取舍和教训

### 设计原则
- AI原生：以AI为中心、以AI全自动化为目标，人类只是临时的协调器。例如，
    - 规则：人类输入原始规则描述，经AI思考加工后再回填 —— AI回填的规则才是有效规则 
    - 需求：人类输入原始需求描述，经AI理解后生成正式输入 —— AI理解的需求才是有效需求(PRD)
    - 知识：知识是AI的知识，人类的感受无关紧要 - AI理解的知识才是有效知识
    - 治理：治理是一种非功能性需求
- AI治理：AI维护项目会发生蜕化，迭代越多项目越大、蜕化越严重，必须自主治理。包括，
    - 方式：治理必须由AI执行(AI治理)，触发机制可人工或自动(受限于Token成本)
    - 对象：代码，知识，产品；技能
- 个性化原则：在通用标准之上，每个Harness实例都有个性化知识，特别是看似通用的Agent、Skill
- 大任务分解：分段Phase、分步Step、分批Batch，子任务
- 上下文管理：在大任务分解的基础上，自动压缩，检查点、文件最小化
- 上下文隔离：不同任务使用不同的Chat窗口，因为上下文污染影响Coding效果
- 多任务并行：低频场景、无需支持。随着AI演进，多人协作开发同一个模块Repo将更低频，现行的worktree + Chat/IDE多窗口只是临时态

### 一些教训
- PRD才是真知识
    - 项目Setup阶段，设计文档是第一次大规模的PRD输入；
    - 项目迭代阶段，①Spec：用户输入模糊的原始Spec，经多轮`AI理解+人机交互`，形成能指导AI的有效输入；②聚化：定期整理琐碎的Spec，回馈到产品的知识体系 => **迭代阶段的Spec形成、聚化**，是本次Harness探索的主要收获；
    - 相比而言，AI自动生成的知识，更多是为了适配AI技术能力，算不得领域知识
- Harness应保留模糊性
    - AI时代，模糊性应被视作一种优点，至少合理范围的模糊性应该；
    - 过度精确，会限制AI发挥，也会失去可迁移性。
    - **过度追求规则精确、过分执着于遵守Workflow**，是本次Harness探索的主要教训；
